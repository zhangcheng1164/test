开场：日志分析的前世今生

8 min

0.1 一个真实的安全事件

🔎

2014 年，某全球知名零售商的 POS 系统被植入恶意软件，攻击者在长达 19 天内窃取了 7000 万条客户信用卡信息。安全团队的 SIEM 系统其实产生了告警——但淹没在每天数百万条日志中，无人处理。

这不是技术失败，而是日志分析能力的失败。

  1. 日志是安全的最后一道防线——攻击者的行为一定会留下日志痕迹
  2. 光收集不够，必须实时分析——T+1 的日志分析在安全场景下等于没有
  3. 需要自动化的规则引擎——人眼不可能盯住每秒数万条日志

0.2 日志分析技术的演进

2000s
grep + awk + cron — 运维手动写脚本，延迟小时级
2010
ELK Stack — 全文检索 + 可视化，但倒排索引占空间大
2015
Kafka + Spark Streaming — 微批处理，延迟 > 100ms
2018+ ⭐
Vector + Kafka + Flink + ClickHouse — 秒级检测 + 亚秒级查询

0.3 今天的培训场景

构建 SSH 暴力破解实时检测系统：1 分钟内同一 IP 登录失败超过 5 次 → 告警。

3.6 Event Time vs. Processing Time

事件时间 (Event Time)处理时间 (Processing Time)
定义日志里的 timestamp算子收到的墙上时间
准确性高（重放一致）低（受网络抖动）
需要Watermark 机制无
本实验简化未用✓ 采用
Watermark 处理乱序数据 
sequenceDiagram
  participant S as 数据源
  participant F as Flink Operator
  participant W as Window [0,10)

  S->>F: Event(t=10)
  S->>F: Event(t=12)
  S->>F: Event(t=9) ⚠️ 乱序!
  S->>F: Event(t=15)
  Note over F: Watermark = 15 - 5 = 10
  F->>W: 触发窗口计算
  Note over W: t ≤ 10 的事件都到齐了
⚠️
生产环境务必用 Event Time + Watermark：WatermarkStrategy.forBoundedOutOfOrderness(Duration.ofSeconds(5))

// SshBruteForceJob.java — 核心 DAG
parsedStream
    .filter(e -> e.getMessage().contains("Failed password"))
    .map(e -> extractIp(e.getMessage()))
    .keyBy(ip -> ip)
    .window(TumblingProcessingTimeWindows.of(Time.minutes(1)))
    .process(new BruteForceDetector())  // count ≥ 5 → 告警
    .addSink(new ClickHttpSink<>(...));
// BruteForceDetector — 窗口处理函数
public void process(String ip, Context ctx, Iterable<String> elements, Collector<AlertEvent> out) {
    int count = 0;
    for (String ignored : elements) count++;
    if (count >= 5) {
        out.collect(new AlertEvent(..., ip, count, ...));
    }
}

4.4 本项目表设计

CREATE TABLE train.raw_logs (
    timestamp  DateTime64(3),
    host       String,
    severity   LowCardinality(String),  -- 字典编码
    facility   LowCardinality(String),
    program    LowCardinality(String),
    pid        UInt32,
    message    String
)
ENGINE = MergeTree()
PARTITION BY toDate(timestamp)   -- 按天分区
ORDER BY (host, timestamp)       -- 排序键 = 主键
TTL toDate(timestamp) + INTERVAL 7 DAY;
⚠️
最大的坑：不要单条 INSERT！每次 INSERT 创建一个 Part，1000次/秒 → Too many parts 错误。正确做法：攒批写入。

4.6 现场 SQL 演练

-- 进入客户端
docker exec -it train-clickhouse clickhouse-client --password train123

-- 各严重级别分布（带 ASCII 柱状图）
SELECT severity, count() AS cnt, bar(cnt, 0, 1000, 40) AS bar
FROM train.raw_logs
WHERE timestamp > now() - INTERVAL 5 MINUTE
GROUP BY severity ORDER BY cnt DESC;

-- SSH 失败 Top IP
SELECT extractAll(message, 'from ([\\d.]+)')[1] AS ip,
       count() AS fail_count
FROM train.raw_logs
WHERE timestamp > now() - INTERVAL 5 MINUTE
  AND message LIKE '%Failed password%'
GROUP BY ip ORDER BY fail_count DESC LIMIT 10;

-- 告警记录
SELECT alert_time, source_ip, fail_count, message
FROM train.alert_logs ORDER BY alert_time DESC LIMIT 20;

-- 压缩率
SELECT table,
  formatReadableSize(sum(bytes_on_disk)) AS disk,
  formatReadableSize(sum(data_uncompressed_bytes)) AS raw,
  round(sum(data_uncompressed_bytes)/sum(bytes_on_disk),2) AS ratio
FROM system.parts WHERE database='train' AND active GROUP BY table;

Q&A 常见问题

Q1: Flink 和 Spark Structured Streaming 怎么选？

延迟 < 1s、复杂有状态计算 → Flink。批流统一 SQL、已有 Spark 生态 → Spark。

Q2: ClickHouse 能做 UPDATE 吗？

ALTER TABLE ... UPDATE 是异步 mutation，重写整个 Part。适合低频修正，不适合 OLTP。

Q3: Kafka 消息会丢吗？

三道防线：Producer acks=all + Broker min.insync.replicas=2 + Consumer 手动提交 offset。

Q4: Flink 挂了状态还在吗？

开了 Checkpoint + 持久化后端，重启自动恢复。Kafka offset 回退 + State 恢复 = 不丢不重。

Q5: 为什么不直接用 ClickHouse Kafka Engine？

Kafka Engine 只能简单入库。窗口聚合、状态机、动态规则等复杂流式计算需要 Flink。
